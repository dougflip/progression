Single-track looper — spike

Record mic input, loop it back with the existing progression. Optimized to ship something usable fast; architecture notes captured for later, not fully solved now.

Goals of this spike

This pass has two distinct goals, and they pull in different directions — worth naming explicitly:

1. Working software, deployed, opt-in

Something the user can actually use behind a feature toggle soon. Major tradeoffs are fine to get there — correctness of the long-term architecture is explicitly not the bar for this pass.

2. Learn what "the right way" looks like

Recording timing/quality in practice, where this code should live, and how the data model should evolve once multiple loops, per-section loops, and per-section playback overrides enter the picture. These are captured as Learning questions below — notes to make while building, not things to build now.

Every shortcut taken to hit goal 1 should be named as it's made — see the Tradeoffs log, which starts empty and fills in during implementation.

Spike scope

1. Ships behind an opt-in toggle — off by default. See Feature flag.
2. Exactly one loop, no overdub. Delete-and-rerecord is the only "edit" — no undo, no layering.
3. Loop length is always the full progression (one full lap) — no configurable loop-length picker this round. That was designed in the previous pass (divisor-of-total-bars, count-in etc.) but adds real complexity (a dedicated per-bar clock, a picker UI) for a question we don't have data on yet. Deferred to Learning question 3.
4. UI is intentionally raw — no polish pass. Three capabilities only: record, play back with the progression, delete.
5. Recording is quantized to the progression's own lap boundary (bar 1 of the chord grid), not a fixed pre-roll length — arming mid-progression just lets the current lap finish as the count-in. Corrected from an earlier "fixed 1-bar pre-roll" design; see Sync mechanism.
6. Optional "mute during recording" toggle, for players without headphones — see Mute during recording.
cycle === 'none' only — unchanged from before. Every lap being a different key still makes a fixed loop clash against a shifting key.
Loop now has its own volume/mute in the Mix sheet — see Loop mixer channel. Originally scoped out ("routes straight to master, no separate slider") but added once the rest of the spike felt solid enough to be worth controlling independently.

Sync mechanism

Recording is quantized to the progression's own lap boundary (bar 1 of the chord grid), not a fixed-length pre-roll. If you arm mid-progression, the current lap simply finishes playing (acting as the count-in — could be anywhere from 1 bar to a full lap depending on when you armed), then recording starts exactly when the progression restarts, so the recorded loop's bar 1 always lines up with the chord grid's bar 1. (An earlier pass tried a fixed 1-bar pre-roll instead, which needed a separate per-bar clock — see the tradeoffs log for why that was wrong and got reverted.)

This reuses the lap-boundary check that already exists inside _buildPart (src/progression-audio.ts, the transportLapIndex > _currentLap block — the same one that advances the key cycle). It already hands us a Tone.js time value — the precise Transport-clock timestamp of "a full cycle just restarted." No dedicated clock needed; the looper piggybacks on the same mechanism that already rebuilds fresh on every start(), so there's nothing extra to keep in sync across stop/start cycles.

Tempo changes after a loop is captured degrade gracefully (truncate a little early or leave a trailing gap) rather than drift, because of the forced-restart trick above. No need to block tempo edits.

Capture

  • Tone.UserMedia for mic input (wraps getUserMedia) → Tone.Recorder (thin wrapper over MediaRecorder). Both ship in the installed Tone.js (15.1.22); zero existing usage in this repo today.
  • Recorder start/stop aren't sample-accurate (Tone's own docs say so — it's just MediaRecorder underneath), so we schedule them via Tone.Draw.schedule(fn, time), the same idiom already used for onChordTick/onBarTick. A few ms of jitter on the recorded content is fine since playback is always re-anchored to the Transport regardless.

Post-record processing

Decode the blob (decodeAudioData), then trim or zero-pad the result to exactly _songBars bars worth of samples at the current tempo. Absorbs MediaRecorder's imprecise start/stop timing so the buffer's length always matches what one lap expects.

Mute during recording

For players without headphones: an optional toggle that silences the chord/bass/drum audio during the recording window only, while keeping the progression fully running underneath — beat dots, chord chip highlights, and the Transport clock itself are untouched, since those are driven by ticks, not audio output.

No new engine surface needed — AudioEngine already has setMute(channel, muted). Sequence:

  1. Before muting, core already knows the current per-channel state from state.mix (chordsOn/bassOn/drumsOn) — no new getter required on the engine.
  2. On entering recording (if the toggle is on): call setMute('chords', true), setMute('bass', true), setMute('drums', true) directly — a transient override, not a state mutation, so it doesn't touch AppState, doesn't re-render, doesn't hit the URL.
  3. On leaving recording: call setMute again per channel using the value already sitting in state.mix, restoring exactly what the user had before (respects a channel the user had already muted manually going into the recording).

Manual sync offset

Real-world testing surfaced audible drift — recorded content lands slightly early or late relative to the beat, from mic/input-chain latency (ADC, OS audio stack, sometimes Bluetooth) that varies per device and isn't something this app can measure. Waveform/beat-detection analysis could estimate it, but that's real complexity for uncertain payoff. Simpler and matches the spike's philosophy: a manual "Loop sync offset (ms)" field in Setup that the player dials in by ear once per device, the same idea flagged as a fallback back when this was first being designed.

Implementation: the full decoded recording is kept around as _rawLoopBuffer (not just the trimmed/aligned version), so nudging the offset re-derives the loop buffer from the original decode without re-recording — the point of a "tweak it by ear" control is fast iteration, and re-recording every time you change the number by 5ms would defeat that. A single pure function does the alignment math: 

alignAndTrimSamples(data: Float32Array, targetLength: number, offsetSamples = 0): Float32Array

Positive offsetSamples skips that many samples off the front of the source (compensates for content that arrived late — shifts the loop's content earlier); negative pads the front of the output with silence (shifts content later). Zero offset reduces to the original trim-or-pad behavior. Unit tested directly (plain arrays in, no AudioBuffer needed) — trim, pad, both offset directions, and the fully-out-of-range case.

AudioEngine.setLoopOffsetMs(ms) stores the value and, if a loop is already captured, immediately re-applies it to _rawLoopBuffer — takes effect on the very next lap's force-restart, no re-arm needed. The Setup field is localStorage-persisted like the other looper settings, so a dialed-in value survives reloads.

Feature flag

A toggle in its own "Looper (Experimental)" Setup-sheet section (split out from the App group once it grew a few controls), persisted the same way theme/keep-awake already are — an app-level, localStorage-backed setting, not part of AppState/URL serialization. Defaults off. When off, the loop button/UI doesn't render at all (not just disabled) — keeps the raw v1 UI from being seen by anyone who hasn't opted in.

Where the state lives

Looper state (idle | arming | recording | looping) is engine-only, never part of AppState — same precedent as _currentLap, _pendingJump, etc. (closure-only in progression-audio.ts, exposed via narrow getters like isPlaying()). Session-only: never serialized to the URL or presets.

Still-arming/recording state is discarded on stop()/rebuild() (mid-capture is meaningless once the transport resets). A completed loop (looping) survives every other structural/settings change, including edits (sections/arrangement) that change _songBars — see Persistence. No auto-invalidation on mismatch: this was tried and deliberately removed (see tradeoffs log) in favor of trusting the player to manage it, matching how tempo-change mismatches already degrade gracefully rather than being blocked.

Persistence

The loop survives a page reload — deliberately scoped to "the current loop persists" rather than a loop library. IndexedDB, not localStorage: the recorded blob is binary and can be a few hundred KB, past localStorage's practical string-only limit. Single fixed record (key "current"), no per-progression or per-preset scoping.

  • Saved (the original recorder Blob, not the decoded buffer — much smaller) right after a successful recording, alongside the trim/decode step already happening in _processRecordedBlob.
  • Cleared on deleteLoop().
  • restoreLoop() — a new AudioEngine method — decodes the persisted blob and sets state to looping at boot, unconditionally: no check that it matches the currently-loaded progression. The player explicitly asked for this — wants to be able to switch progressions and come back to find the loop still there, and is fine managing any mismatch themselves.
  • One wrinkle: at boot, _songBars/_channels aren't known yet (nothing's played), so the restored buffer can't be trimmed into a Tone.Player immediately. restoreLoop() only stores the raw decoded buffer; the actual Player gets built the first time start() runs afterward, reusing the exact same "restore" branch that already existed for resuming a same-session loop after stop→start.
  • App boot calls app.restoreLoop() once, then manually syncs the loop button's label — the state change happens before onLooperStateChange is wired up (that only happens inside start()), so the callback alone wouldn't reach the UI in time.

Loop mixer channel

Independent volume/mute for the loop, in the Mix sheet — gated by the same looperEnabled flag as the record button (not by cycle === 'none', since a captured/restored loop keeps sounding regardless of cycle mode — that restriction only ever applied to arming a new recording).

Straightforward: reuses the exact pattern already in place for chords/bass/drums — a 4th Tone.Channel (loop, connected to master alongside the other three), with _loopPlayer connected to it instead of directly to master. setVolume/setMute both grow a "loop" case on the existing channel union, so no new methods were needed on AudioEngine itself — just two new core wrapper methods, setLoopVolume(v)/setLoopMuted(m), that call straight through to audio.setVolume("loop", v)/audio.setMute("loop", m) — bypassing setMix()/AppState entirely, same "engine-only, not persisted in the URL/presets" treatment as the rest of the looper.

One wrinkle mirroring the sync-offset field: the slider/checkbox need to work at boot, before any channel exists (mirrors the same _songBars-not-known-yet problem Persistence hit). setVolume/setMute now store the loop's volume/mute in a closure var before checking whether _channels exists (chords/bass/drums keep their original early-return, unchanged) — a _syncLoopMixToChannel() call inside _initChannels() applies whatever was stored the moment the channel is actually created. Persisted via localStorage, same as the sync offset.

Verification

This is fundamentally live-audio — headless e2e can't meaningfully exercise it.

  1. npm run check + npm run test:quiet — typecheck, plus unit coverage for the buffer trim/pad math (extracted as a plain function over sample counts, testable without a real AudioBuffer).
  2. Manual testing (project convention — the user drives this). Scenario list once built:
    • flag off by default — confirm no loop UI appears anywhere
    • flag on, happy path — arm → count-in → record → loop plays back in sync
    • mic-permission-denied path
    • mute-during-recording toggle on — confirm silence during capture but beat dots/chip highlights still animate live, and prior manual mute state is restored correctly afterward
    • arming before playback has started
    • tempo change after a loop is captured (confirm graceful truncate/gap, not drift)
    • stopping playback mid-recording (confirm clean cancel/discard)
    • deleting a loop and recording a fresh one in its place
    • switching away from cycle: 'none' while a loop exists
    • reload the page after capturing a loop — confirm it's still there and plays in sync (once start() runs) without re-recording
    • change the progression's bar count, then reload — confirm the mismatched loop still restores rather than being silently dropped
    • Mix sheet Loop row — mute/unmute and drag the volume slider while the loop is playing; confirm the row only appears when the flag is on, and the value survives a reload

Learning questions

Notes to make while building, not specs to implement now. Revisit this list once the spike is in hand.

1. How well does the device actually record, timing-wise?
Watch for: real-world MediaRecorder start/stop jitter across browsers/devices, and whether the trim/pad step is enough to hide it or whether the loop audibly stutters at the restart point. Quality (mic, codec) is largely out of our hands — timing is the thing we can actually act on.

Answer so far: not perfectly — real testing found consistent audible drift (input-chain latency), fixed for now with the manual sync offset field rather than automatic detection. Worth revisiting whether that latency is roughly constant per-device (a one-time dial-in is enough) or variable enough that a fixed offset won't hold up over longer sessions.
2. How should the looper code be organized?
Options: folded directly into progression-audio.ts (fastest for the spike, what this doc currently plans) vs. a new dedicated module (e.g. progression-looper.ts) that progression-audio.ts composes, mirroring how audio itself is a swappable dep behind the AudioEngine interface. Building it inline first, then watching how much it entangles with existing playback code, is probably the fastest way to find out whether it deserves its own module.
3. How would the data model change for multiple/independent loops?
Shape of the question: today's spike has exactly one loop, engine-only, never in AppState. Real overdubbing needs loops to be addressable — mixed independently, muted independently, maybe reordered/deleted independently. That likely means loops do need to become first-class state (if only for UI to list/manage them), even if the audio buffers themselves stay off to the side (probably too large for AppState/URL serialization the way chords are). Also ties into whether loops belong to the whole song or to a specific section (see next question) — worth deciding those together rather than bolting section-scoping on after the fact.
4. Are per-section loops and per-section playback overrides the same underlying need?
Yes, probably the same shape. "Section 2 has its own loop" and "section 2 plays a different style/bass variant" are both "this section carries an optional override/extra payload beyond the base song settings." Worth designing one mechanism — likely something like an optional Partial<PlaybackSettings> merge per section, already flagged as a candidate in docs-internal/feature-ideas.md under "Section-level playback overrides" — and letting loops hang off the same per-section extension point rather than inventing a second, parallel mechanism.
5. How do we manage multiple loops in the UI?
Likely needs more surface than a single button. Once there's more than one loop (per song or per section), the player needs to see what exists, mute/solo/delete individually, and probably see which section a loop belongs to. That's real UI, not a button-label swap — plausibly its own screen/sheet once it gets here, unlike this spike's single button.
6. Does a "build your own drums+bass part" feature imply three screens (playback / loop / drum-bass builder)?
Plausible, not yet decided. If both loop management and a drum/bass builder turn out to need real screen space, the app's shape might become: a main playback screen (as today), a loop management screen, and a drum/bass builder screen — each addressing a different "extra layer" on top of the base progression. Worth revisiting once question 5 has an answer, since loop management is likely to need dedicated space first.

Tradeoffs log

Filled in during implementation — every shortcut taken to hit "working software fast" gets a line here, so it's not lost by the time we revisit the real architecture.

Sync mechanism churned through three iterations in one manual-testing pass — worth recording the sequence, since each step was a genuine correction, not just polish:
  1. v1 — reused the lap-boundary check for every transition. Bug (found via manual testing: "Get Ready…" never advanced): with a fixed-1-bar-pre-roll requirement, that check only fires once per full lap, so arming and pre-roll each took a full lap instead of 1 bar.
  2. v2 — added a dedicated per-bar Tone.Loop clock to get real bar granularity. Fixed the wait-time bug, but introduced a second one: stop()'s Tone.Transport.cancel() silently wiped the clock's schedule, so a loop that survived a stop→start cycle would stop retriggering. Fixed by explicitly recreating the clock in start().
  3. v3 — the requirement itself was wrong, not just the mechanism: a fixed 1-bar pre-roll was never what was wanted. Recording needs to quantize to the progression's own lap boundary (bar 1 of the chord grid) so the loop always lines up with the chords, with arming-mid-progression simply riding out the current lap as a variable-length count-in. That's back to exactly v1's mechanism — which makes v2's per-bar clock (and the bug it introduced) entirely unnecessary. Net result: simpler code than v1 started with, once the per-bar clock was removed again.
stop() didn't actually stop loop playback — a real bug, found via manual testing (loop kept playing for up to a full lap after hitting Stop). Stopping the Transport doesn't stop an already-started Tone.Player; it just plays out to the end of its own buffer. Fixed with an explicit safeCall(_loopPlayer, "stop") when stop() is called while looping — state stays looping (the captured loop is still valid), only the audible playback is cut.
Any rebuild() cancels an in-progress capture, not just structural edits that change _songBars. Changing drum style mid-count-in discards the capture rather than trying to carry it through. Simple; a real implementation might want to preserve capture across a style/bass/voicing tweak.
Fixed: loop playback used to take one extra silent lap to resume after stop→start. Found via manual testing ("why does resuming need a delay if the buffer's already decoded?"). The forced-restart trigger only fires when transportLapIndex > _currentLap, which is false on the very first lap of a fresh start() (by design, so key-cycle advance doesn't fire early either) — so a completed loop sat silent for one lap before retriggering, even though nothing was actually pending. Fixed with an explicit immediate-trigger-on-start() when a loop already exists, on top of (not instead of) the per-lap force-restart that keeps it in sync going forward. This one only applies on resume — the very first playback after a fresh recording still waits for decode, which is a real constraint, not an artifact.
Mic stream and Recorder are never explicitly closed for the page session — only the loop buffer/player are disposed on delete. Avoids re-prompting for mic permission on every re-arm, at the cost of holding the input stream open longer than strictly needed.
No pulsing/color styling for recording state — just a plain label swap on the existing .bar-btn, per the "raw UI" scope decision. Easy to layer on later.
Removed rebuild-time auto-invalidation of a mismatched loop — the player explicitly asked for this: wants to switch progressions and come back to find the loop still there, fully willing to manage a mismatch themselves. The removed code was defensive but unnecessary: the force-restart-every-lap mechanism already degrades gracefully (truncate/gap) for a tempo mismatch, and the same reasoning covers a bar-count mismatch just as well — nothing new needed, just less code.

Final UI decision

Single toggle button (idle → record → loop → delete). Delete goes through a plain confirm() browser dialog to guard against an accidental tap — cheap enough for a spike, revisit if it feels wrong.
