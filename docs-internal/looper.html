Looper

Record mic input, loop it back against the running progression. Ships opt-in today as a single fixed loop; the next phase is multiple layered loops per section — see Multi-loop data model.

Background

Lets a player record themselves (or an instrument) over the progression and hear it loop back on top of the chords/bass/drums — practicing over your own take, not just the backing track. Opt-in via a Setup toggle, off by default, and currently only available when cycle === 'none' — a fixed-length loop would clash against a key that shifts every lap.

Engine lives in src/progression-audio.ts, exposed through the AudioEngine interface in src/progression-core.ts, with UI/state glue in src/app.ts — same three-file split as the rest of the app (see ARCHITECTURE.md).

What's built

Record / play / delete

Mechanics below reflect 2a/2b (see Multi-loop data model) — the user-facing feature is unchanged for a single-section song, but the underlying sync boundary and state machine are not what they were pre-2b.

One loop per section, no overdub — delete-and-rerecord is the only edit. State machine idle → arming → recording; "does section X have a loop" is a derived read (sections[X].loops.length > 0), not a state value. Driven by section-entry (ev.chipIndex === 0 in _buildPart's Tone.Part callback) rather than the whole-song lap boundary — arming rides out whatever's left of the current section as a variable-length count-in (landing on whichever section's own bar 1 comes next, not necessarily the one active when armed); recording finalizes on that same section's next entry, so the loop's bar 1 always lines up with its owning section's bar 1.

Playback force-restarts on every section-entry — player.stop(time); player.start(time), same pattern as drum sample triggering (_triggerDrum) — no Player.loop/.sync(), so it can never drift relative to the Transport. A tempo change after capture degrades gracefully (truncates a little early or leaves a trailing gap) rather than drifting. One shared Tone.Player serves whichever section is currently active, swapping buffers by loop id as sections change — a gap on cache miss is accepted for now (see 2b's storage/swap decisions below).

Capture: Tone.UserMedia → Tone.Recorder, scheduled via Tone.Draw.schedule. Post-record: decode, then trim/zero-pad to exactly the recording section's own bar length (_sectionBars[sectionIndex]).

Mute during recording

Optional toggle for players without headphones: silences chord/bass/drum audio for the recording window only (beat dots/chip highlights/Transport keep animating — they're tick-driven, not audio). A transient setMute override per channel, restored to whatever state.mix already had afterward — never touches AppState.

Manual sync offset

Mic/input-chain latency varies per device (ADC, OS audio stack, sometimes Bluetooth) and isn't something the app can measure — a "Loop sync offset (ms)" field in Setup lets the player dial it in by ear once. Measured range so far: ~50ms on a laptop, ~350ms on a phone — wide enough that this has to stay per-device/manual.

Each loop's decoded recording is cached by id (_loopBufferCache), so nudging the offset re-derives the trimmed buffer without re-recording, via a pure function: 

alignAndTrimSamples(data: Float32Array, targetLength: number, offsetSamples = 0): Float32Array

AudioEngine.setLoopOffsetMs(ms) immediately re-trims whichever loop is currently loaded into the shared player; other sections' loops pick up the new offset lazily, next time they're swapped in (_swapToSectionLoop always uses the current _loopOffsetMs). Still a single global/per-device value, not per-loop — it corrects mic/input-chain latency, a device property, not a content property. Persisted in localStorage like the other looper settings.

Persistence

Metadata (LoopRef, including the loop's id) lives on Section.loops[0] and rides through the normal AppState/preset boot path — no dedicated restore step. The audio blob itself stays in IndexedDB, one row per loop id (keyPath: "id", not localStorage — blobs are binary, a few hundred KB each). Whichever section is entered during playback fetches its own blob by id (cached in memory after the first fetch) — no check against the currently-loaded progression, same "the player manages any mismatch themselves" philosophy as before, just applied per-section now. Deleting a section's loop (deleteLoop(sectionIndex)) removes its IndexedDB row; other sections' loops are untouched.

Mixer & cleanup effects

A 4th Tone.Channel (loop) gives the loop independent volume/mute in the Mix sheet, gated by the feature flag rather than cycle === 'none' — a captured loop keeps sounding regardless of cycle mode; that restriction only applies to arming a new recording. Three optional cleanup effects sit between the player and that channel, chosen for common mic problems rather than as creative effects:

  • Tone.Filter (fixed 90Hz highpass) — cuts mic rumble/handling noise
  • Tone.Compressor — single 0-100 "amount" slider mapping threshold+ratio together
  • Tone.Limiter (fixed -3dB) — catches occasional clipping

Highpass/limiter are true bypasses — physically removed from the graph when off, not just neutralized — so toggling one is a clean A/B. _rewireLoopChain() rebuilds _loopPlayer → [highpass?] → compressor → [limiter?] → loop channel on every toggle.

Feature flag & UI

looperEnabled, a localStorage app-level setting (same shape as theme/keep-awake) in its own "Looper (Experimental)" Setup section — defaults off, and the loop UI doesn't render at all when off (not just disabled). One button (#loop-btn) in the action bar, label combining LooperState with whether the active section already has a loop: idle+no-loop "⏺ Record Loop", arming "⏳ Get Ready…" (tap = cancel), recording "● Recording…" (tap = cancel), idle+has-loop "🗑 Delete Loop" (behind a confirm() dialog). Hidden when the flag is off or cycle !== 'none'. Tapping "Record Loop" from a full stop starts playback automatically first — no separate press-play-then-record step.

Multi-loop data model

Unreleased — backward compatibility is not a goal, for any phase of this build. Only the developer is dogfooding this; nobody else has recorded loops that need to keep working. If a cleaner shape for the current phase or a later one requires breaking the stored data format, an existing setter signature, or in-progress recordings, just break it — bump the IndexedDB version and wipe the store, change a method signature, whatever's cleanest. Don't spend effort on migration code, optional/fallback fields, or preserving old behavior "just in case." Optimize purely for the best shape to land on across Phases 1-3.

where we're headed — 2a/2b implemented, 2c/2d designed but not built

What it takes to go from today's single fixed loop to multiple loops per section, layered together. 2a/2b have landed; 2c/2d are still just the shape agreed on before touching anything. 

interface LoopRef {
  id: string;            // uuid — IndexedDB key for the raw recorded blob
  label?: string;        // optional; UI falls back to "Loop N" by array position
  capturedBars: number;  // this section's bar length at capture time
  volume: number;        // 0-100, per-loop
  muted: boolean;
  compression: number;   // 0-100, per-loop
  highpass: boolean;
  limiter: boolean;
}

interface Section {
  progression: string;
  loops: LoopRef[];      // 0+, ALL non-muted loops play together
}
Layered, not single-active-loop. Every non-muted loop on a section plays at once — a loop-pedal feel, not a track picker.
"Mute while I record" extends to sibling loops. Playback during recording is otherwise exactly what the user has configured — if a layer is audible, it stays audible while you record on top of it, and muting one before you arm is the way to keep it quiet. But the existing mute-during-recording toggle (see Mute during recording), which today only silences chords/bass/drums, grows to cover loops too: checking it silences all audio — chords, bass, drums, and every existing loop — for the recording window, same transient-override-then-restore treatment already used for chords/bass/drums (never touches each loop's own muted flag in AppState).
Sync boundary moves from whole-song lap to section entry. Per-section loops need to restart every time that section starts, which can happen more than once per lap (e.g. section 1 in arrangement "1 2 1 2"). As landed in 2b, the signal for this is ev.chipIndex === 0 inside _buildPart's Tone.Part callback (see Record / play / delete) — every section's first chord always has chipIndex === 0, whether it's a genuinely new section or a manual-hold repeat of the same one. ChordTickEvent.sectionChanged/sectionIndex (the original plan here) turned out to be the wrong signal to reuse: it's Draw-scheduled/UI-facing (not precise enough for audio scheduling) and deliberately doesn't re-fire on a manual-hold repeat of the same section — exactly the case a loop restart needs to fire on. Side effect: every occurrence of a repeated section replays the same loop by default, which naturally answers the per-instance-loop question without having to solve it yet — a later per-instance override would layer on top of this, not replace it.
LooperState narrows to idle | arming | recording. "looping" drops out as a global state — whether a loop exists is now per-section and many-valued (section.loops.length > 0), not one app-wide boolean. "Something is looping" becomes a derived read, not a state machine value; the state machine only needs to describe the capture workflow itself.
Storage split is unchanged in principle, multiplied in practice. LoopRef metadata (small, JSON) lives on Section, rides along in presets/AppState for free. Audio blobs stay in IndexedDB, keyed by id instead of today's fixed "current" record (see Persistence). The manual sync offset stays a single global, per-device setting rather than per-loop — it corrects mic/input-chain latency, a device property, not a content property — and today's design already stores raw blobs and derives trimmed buffers at load time, so this generalizes with no change needed.
Local-only, deliberately. Loops stay out of URL sharing — binary audio can't travel in a URL — so a shared link carries chords/arrangement as it does today, silently without loops. Presets can carry loop metadata, but become effectively device-local for loop content: a preset's loop ids are meaningless without matching IndexedDB rows on that browser. First time URL-shareable state and preset-saveable state diverge.
Garbage collection by periodic sweep, not cascade-on-delete. A saved preset can still reference a loop id no longer in live state, so deleting from live state can't blindly drop the IndexedDB row. Instead: collect every loop id referenced by {live state} ∪ {every saved preset}, delete anything in IndexedDB not in that set, run the sweep at safe checkpoints (after deleting a loop/section/preset) rather than reference-counting on every mutation.
Deferred: section duplication. Not a planned feature today, so the question of what happens to a duplicated section's loops (start empty / deep-copy into new IndexedDB rows / alias the same ids) is left open — it doesn't change the data structures above either way, only how a future duplicateSection() would populate them. Revisit only if that feature actually gets proposed.

Rollout phases

Sequenced so each phase — except the last — ships with no visible UX change, verified before moving to the next.

The "Phase 2" bullets immediately below describe its target design as a whole, written before it was split into stepping stones — some of what they describe has landed (2a/2b, below), some hasn't yet (2c: confining playback during recording; 2d: mixer scoping). See "Phase 2 sub-phases" further down for what's actually built right now.

Phase 1 — arrayify, engine-only. Reshape the single loop's metadata into _loops: LoopRef[], capped at length 1 by today's existing state-machine gating (arm still blocked once one exists). Still one shared Tone.Player + effects chain — the array only changes how metadata is held/iterated. No AppState/Section involvement yet, no UX change.
Phase 1 implementation. LoopRef stays local to progression-audio.ts — it only moves to progression-core.ts when Section needs it (Phase 2). AudioEngine's public surface is unchanged; setters keep today's flat signatures and target _loops[0] internally. id (crypto.randomUUID()) is generated per loop and persisted in the IndexedDB record now — not because Phase 1 needs it for anything, but it's free and gets Phase 3's id-keyed rows closer. Since the feature is unreleased/experimental, the schema change bumps LOOP_DB_VERSION and wipes the old object store rather than writing migration/fallback code for the old shape — no existing data is worth preserving compatibility for.
Volume/mute/compression/highpass/limiter default from a small _loopDefaults bucket, not per-loop persistence. These settings still live in app.ts's localStorage keys, applied via the same setter calls as today. Those setters always write into _loopDefaults (five fields, not a full LoopRef) and additionally mirror into _loops[0] when a loop exists; a newly captured or restored loop seeds from _loopDefaults. Not a compatibility shim — it's what keeps "record, delete, re-record" and "reload the page" behaving sensibly while this is actively dogfooded through the next few phases, with zero app.ts changes needed.
Phase 2 — attach to a section, fix the sync boundary. loops moves onto Section/AppState; the restart trigger switches from whole-song lap to section-entry (ev.chipIndex === 0, as landed — see the sync-boundary note above); capturedBars means the owning section's bar length, not _songBars; LooperState narrows to idle | arming | recording. Invisible for a single-section song (section-start == song-start); a real, accepted behavior change for a multi-section arrangement with an existing loop — its restart cadence/length changes the moment this ships. Flagged, not mitigated, since the feature is still experimental.
Phase 2 also allows one loop per section, not one loop per song. A multi-section song can carry a different loop on each section — recording while section 2 is active attaches the loop to section 2, independent of whatever section 1 already has. Because sections play sequentially, never simultaneously, at most one section's loop is ever audible at a time, so this still needs only a single shared Tone.Player + effects chain, reloaded with whichever section's buffer is active — not the N-concurrent-player setup Phase 3 needs for actual layering.
Phase 2 also confines playback to the target section while recording. Arming a recording switches the transport to loop just that section — its own chords/bass/drums, repeating — instead of continuing through the full arrangement. This holds through arm → record → looping, so auditioning and re-recording a take happens against a tight loop of just that section, not the whole song; pressing Stop returns to normal full-arrangement playback. Implemented by treating the target section as the transport's loop target for the duration, reusing the existing jump/seek machinery (queueJump/seekToPos) rather than new Part-boundary logic.
Phase 2 also scopes the mixer to the selected section. The Looper group in the Mix sheet reflects whichever section is currently active or selected in the song scrubber — its own volume/mute/compression/highpass/limiter, read from and written to that section's LoopRef rather than a single flat set of engine values. Selecting a different section (the same scrubber tap already used for playback position) brings that section's loop into the mixer; if the selected section has no loop, every control in the group is disabled/dimmed rather than hidden. Per-loop setter methods (setLoopVolume, setLoopMuted, etc.) take a section reference and route through the same updateSection-style mutation as other section edits, rather than the engine-only bypass the single-loop version used.
Phase 2 also makes cross-song persistence work for free. "Song" here means preset — the only named, switchable saved-song concept this app has. Presets already serialize the full sections array (PresetState = Omit<AppState, "activeSection">), so once loops live on Section, saving/loading a preset carries its loops along: record a loop in song 1, switch to song 2 and record a different one, switch back — both are just preset loads, no new mechanism needed for the metadata half. Two things this depends on:
  • The GC sweep must consider loop ids referenced by every saved preset, not just live state — otherwise switching away from a preset would look like an orphan and get swept before you switch back to it.
  • Loops are deliberately excluded from the URL, so a plain reload of an unsaved song won't carry its loop-to-section links the way it carries everything else (sections/arrangement/playback/mix). Accepted as a known limitation rather than a third persistence channel. This is not the same as "reloading a preset" — see the bug/fix noted below for the distinction that actually matters.
Bug found in testing, fixed: a plain browser refresh never actually reloaded a saved preset's loops, even with presetId sitting right in the URL. Since syncUrl keeps the address bar in sync with current state on every render, location.search is non-empty almost immediately after any app usage — so a refresh takes the applyUrl(location.search) boot path, which rebuilds sections from ?section= params (loop-less, since loops were never serialized into the URL). The subsequent presetId handling (setLoadedUserPresetContext/setLoadedBuiltinPresetContext) only tags "you're viewing preset X" for UI/dirty-check bookkeeping — it never re-applies preset.state, so the loops silently never came back. Only an explicit "Load" from the Presets menu (loadUserPreset, which does call _applyPresetState) ever actually restored them — a plain refresh, even of a saved song, did not.

Fixed surgically rather than by fully re-applying the preset on boot: a full re-apply would overwrite any non-loop tweaks made since the last save (currently preserved because the URL wins), trading one loss for another. Instead, a new mergeSectionLoops(sections) backfills only .loops onto the current (URL-derived) sections, matching by index, called right after the existing setLoaded*PresetContext calls in app.ts's boot sequence — everything else the URL already resolved (progression edits, tempo, whatever) is left untouched.

Phase 2 sub-phases

Phase 2 is bigger than Phase 1 — it's split into stepping stones, each independently shippable, rather than landed in one pass. 2a and 2b can't fully separate (the section-entry sync boundary only makes sense once a loop actually belongs to a section), but 2c/2d are additive on top of 2b and can land independently, in either order.

2a — relocate metadata, no behavior change. Move LoopRef from progression-audio.ts into progression-core.ts; add loops: LoopRef[] to Section. The engine still behaves exactly like Phase 1 internally (one shared loop, lap-boundary sync, fixed "current" IndexedDB key) — app.ts just mirrors that single LoopRef onto sections[activeSection - 1].loops after capture/restore. Gets loop metadata riding along in presets for free, with zero new storage or sync work yet.
2b — real per-section cap, section-entry sync, flat id-keyed storage. Lift "1 per song" to "1 per section"; sync boundary moves from lap to section-entry (ev.chipIndex === 0); LooperState narrows to idle | arming | recording — "is section X looping" becomes a derived read (sections[X].loops.length > 0) instead of a state value, since multiple sections can independently have a loop while only one capture session runs at a time. Sequential sections can now each hold a saved loop simultaneously, so IndexedDB needs real multi-row storage — keyed flatly by loop id, not partitioned by song/preset id. Reasoning: the current working state only has a stable id if it was loaded from (or saved as) a preset (getLoadedPresetId(), progression-core.ts) — an unsaved song has none, so song-keyed storage would have nowhere to put a freshly-recorded loop's blob until the first save. A flat loop-id keyspace sidesteps this entirely: "which song owns which loop" lives only in the JSON (Section.loops[].id), never in IndexedDB's structure. This also matches the already-decided global GC sweep (collect every id referenced by live state ∪ every saved preset, delete anything else). Originally this also meant "Save As" could share a loop blob across two presets for free — reconsidered below after testing revealed why that's the wrong default.
Reversed after testing: saveUserPreset ("Save As") deep-copies each loop into a fresh IndexedDB row instead of sharing the original's. The original reasoning (flat loop-id keyspace ⇒ "Save As" shares a blob for free, no duplication) was correct on paper but wrong in practice — forking a song and finding it still carries the old song's loops was surprising, not helpful, and it quietly recreated exactly the shared-reference risk the GC-sweep design was trying to avoid: deleteLoop() deletes its IndexedDB row unconditionally, with no check for whether another saved preset still references that id. Sharing ids across presets meant deleting a loop in the new song could silently break the old song's loop too. (An intermediate fix just cleared loops from the new song instead — simpler, but discarded the audio on every fork, which wasn't the goal either.)

Landed instead: a new AudioEngine.copyLoop(id): Promise<string | null> reads the existing blob by id, writes it back under a fresh crypto.randomUUID(), and returns the new id (null if the source blob is gone — an orphaned reference isn't something to propagate into the new song either). saveUserPreset copies every section's loop this way into both live state and the saved snapshot before persisting, so the session you land in after Save As is already using the new copies, not the old ids — same requirement as the "clear" version had, for the same reason (otherwise editing the new song would still be mutating the old one's blob). This makes saveUserPreset genuinely async (Promise<string>, not string) — app.ts's Save-As handler split into a small async helper (saveNewPreset) rather than making the whole click handler async, since the rename/overwrite branch stays synchronous. overwriteUserPreset ("Save," updating the same song) is unaffected — same song, same loops, no copying. Storage cost: every Save As duplicates the loop blob(s), not just metadata — accepted for now given "a few hundred KB each" scale; revisit with a normalized loop table (many songs referencing one stored blob, reference-counted) only if usage patterns actually produce enough loops for that to matter — premature otherwise. With no two presets ever sharing a loop id, deleteLoop()'s unconditional delete stays safe by construction, no reference-counting needed today.
2b tracks _currentSectionIndex engine-side — no section param on armLoopRecording. The engine has no concept of AppState.activeSection today (that's an app.ts/progression-core.ts idea) — but every chord event already carries PartEvent.sectionIndex (_buildPart), it's just not persisted anywhere the way _currentPosIndex is. So _currentSectionIndex gets tracked the same way, updated from ev.sectionIndex in the Tone.Part callback, and armLoopRecording() takes no section argument. As landed: it still pre-checks _currentSectionIndex at click-time (an early, non-authoritative gate, since Tone.Transport must already be running to arm at all), but the section that actually gets recorded is captured live inside _onSectionEntry at the arming→recording transition — which also re-validates the cap, in case the count-in crossed into a different section than the one active at click-time (see below). Arming rides out a count-in before recording starts, so a passed-in param could go stale; deriving it live can't. capturedBars follows the same logic — needs a per-section bar-length lookup (built in _buildPart alongside the existing posOffsets/chipOffsets maps) instead of the whole-song _songBars.
2b threads sectionLoopIds across the engine/core boundary. The data model itself doesn't change — a section either has a loop id in loops[0] or it doesn't. But the engine never reads AppState directly (by design; it only knows what start()/rebuild() hand it, same as chordSequence/mix/key today), so it needs that fact handed to it explicitly to gate re-recording and to know which buffer to load as sections change. New field sectionLoopIds: (string | null)[], one slot per section, computed as sections.map(s => s.loops[0]?.id ?? null) at the two call sites already constructing engine opts (_startPlayback(), _scheduleRebuild()), plus a lightweight AudioEngine.setSectionLoopIds() to push updates after a capture/delete without triggering a full rebuild().
2b's section-buffer swap is async, with a gap — accepted as a rough-in, not the final answer. On every section-entry (ev.chipIndex === 0), if the new section's loop id differs from what's loaded, fetch-from-IndexedDB → decode → swap into the shared Tone.Player (_swapToSectionLoop). Gapless playback is a hard requirement for calling Phase 2 actually done, but not for landing 2b — ship the gap now, iterate before Phase 2 is considered complete.

Likely fix — prime everything up front, not just "the next one." As soon as sectionLoopIds is known (start()/rebuild()/ setSectionLoopIds()), kick off a background fetch+decode for every id not already in _loopBufferCache, populating the cache before any of them are actually needed. Given the doc's own framing of these blobs as "a few hundred KB each," priming all of a song's loops at once is simpler and more robust than tracking "what's next" — no need to special-case manual mode, jumps, or non-sequential arrangements, where "next" isn't well-defined. A look-ahead approach (prefetch specifically the upcoming section in the resolved play order, during the current section's playback window) would use less memory/bandwidth for a song with many large loops, but that's not this app's likely shape — start with prime-everything, revisit only if it proves too eager in practice.
Fixed: resuming mid-section now resumes the loop mid-loop, not always from its own bar 1. start()'s proactive swap for the starting section computes intoSectionBars = startBarOffset - (posOffsets[startPosIndex] ?? 0) (bars into this section-occurrence, from the same posOffsets/chipOffsets _buildPart already returns), converts it to seconds via Tone.Time, and passes it as _swapToSectionLoop's offsetSeconds argument — mirroring how Phase 1 handled the equivalent song-relative resume case. Computed from the section's own nominal bar length rather than the buffer's actual decoded duration, so it doesn't need to wait on the async cache-miss path to be known; a length mismatch degrades the same "truncate/gap, not drift" way tempo changes already do.
2b's IndexedDB store uses keyPath: "id", in-line keys. createObjectStore(LOOP_STORE_NAME, { keyPath: "id" }), then store.put(record) with no separate key argument, store.get(id) for lookup. One canonical place a loop's identity lives — the record itself — rather than id-as-value plus key-as-separate-bookkeeping. Scales cleanly toward "many loops per section, many sections per song," and store.getAll()/getAllKeys() come along for free for the GC sweep and the manual-cleanup path noted below.
2b also accepts orphan accumulation — no real sweep yet. Multi-row storage means deleted/abandoned loops can leave orphaned IndexedDB rows before a real sweep exists (that's Phase 3's job, see below). Accepted for this unreleased/dogfood-only feature — plan for a manual cleanup path later (a console snippet or a hidden Setup button comparing IndexedDB keys against the union of all saved presets' referenced ids), not an automatic sweep now.
2b removes restoreLoop() entirely — it's obsolete, not refactored. Confirmed via app.ts's boot sequence: if a default preset or presetId URL param is set, AppState.sections[i].loops is already populated through the normal preset-loading path — no special restore step needed, since metadata genuinely lives on Section now (2a). The only thing IndexedDB needs to answer going forward is "does this loop id's blob still exist," which the same fetch-by-id-on-section-entry logic that handles ordinary playback already answers, with no special-casing for "just booted." A plain reload of an unsaved song still won't carry loop-to-section links (accepted limitation, unchanged from before) — but that's a URL/preset-serialization question, not a reason to keep dedicated restore machinery.
2b's deleteLoop() takes a sectionIndex param — unlike armLoopRecording, which correctly has none. Deleting must work regardless of playback state (stopped included), so it can't derive its target from the engine's live _currentSectionIndex the way arming does — it needs AppState.activeSection, which only progression-core.ts has. Distinct from 2d's mixer setters (which take a section reference by design) — this needed to land in 2b or per-section loops couldn't be cleanly tested/managed.
Arming re-validates the per-section cap at the actual transition, not just at click-time. armLoopRecording() only checks the section active when the button is clicked — but the count-in can span multiple sections before the next chipIndex === 0, so the section that count-in actually lands on might already have a loop even though the one active at click-time didn't. Re-checked inside the arming→recording transition itself: if the landing section already has a loop, keep waiting (stay armed) for one that doesn't, rather than recording over it.
2c — confine playback to the target section while recording. Mostly already exists: the manual-advance branch in _buildPart's Tone.Part callback (progression-audio.ts:508-523) already resets Tone.Transport.position back to _currentPosIndex's start on every event whose posIndex differs — a "hold on this section forever" primitive, independent of one-shot jumps. 2c reuses this: on arm, force that same hold onto the target section regardless of the user's actual _advance setting; on cancel-arm, drop the hold and resume normal _advance behavior from wherever the transport naturally is; on Stop, no special-case needed — the existing unconditional reset of _currentPosIndex/_pendingJump already undoes the hold as a side effect. No new Part-boundary logic required.
2d — scope the Mix sheet to the selected section. Piggybacks on AppState.activeSection (progression-core.ts), which already tracks both "currently playing section" (updated on sectionChanged) and "section selected via scrubber tap" — exactly the hook this needs, no new state required.
Phase 3 — lift the one-loop-per-section cap. The actual multi-loop feature: N players/effects chains per section (likely means the looper finally deserves its own module, e.g. progression-looper.ts, rather than staying folded into progression-audio.ts), the Looper mixer group becoming a list of rows — one per loop on the selected section — instead of one, mute-during-recording extended to sibling loops, loops round-tripping through id-keyed IndexedDB rows for real, the GC sweep actually implemented. Deliberately left underspecified — no sense detailing this before phases 1-2 land and inform it.

What this enables

  • Different sections in the same song can each carry their own loop — no longer one loop for the whole song
  • Multiple independent loops stacked on one section — real loop-pedal-style layering, not a single track
  • Per-loop volume/mute/compression/highpass/limiter, tuned by selecting a section in the scrubber and opening the Mix sheet
  • Loops saved and reloaded as part of a preset, alongside the progression they belong to
  • Every occurrence of a repeated section plays the same loop set automatically, no extra sync work
  • Recording plays back just the section being worked on, not the whole song, until you stop
  • A one-click "silence everything" option for recording without headphones, on top of per-loop manual mute

Limitations / tradeoffs

  • No per-instance override — all occurrences of a repeated section share the same loops. Loops still belong to the section either way, not to a specific occurrence — if this is ever needed, it's most likely an additive config layer keyed by arrangement position (e.g. "mute loop X on the 2nd occurrence of section 1"), not a change to Section.loops itself.
  • While recording, you only hear the section being worked on — no live preview against the rest of the song until you stop and press play again
  • Loops don't travel through shared URLs — a shared link carries chords/arrangement only
  • Presets carrying loop metadata are effectively device-local — the audio doesn't move with the preset across browsers/machines
  • Still one capture at a time — layering happens by recording sequentially, not simultaneous multi-track capture
  • Deleted loops' IndexedDB rows aren't reclaimed immediately — cleanup happens on the next GC sweep, not the instant you delete
  • Meaningfully more engine complexity — N players/effects chains to manage instead of one

Other ideas under consideration

Smaller, independent of the data model above — each is a toggle-and-judge experiment, not a commitment.

  • Loop-seam declick — not prioritized, no reported clicking yet; if revisited, snap trim points to the nearest zero-crossing rather than fading amplitude (a fade risks touching beat 1 or sounding like an audible cut on beat 4).
  • Raw mic constraints vs. our cleanup effects — A/B getUserMedia's built-in echoCancellation/noiseSuppression/autoGainControl against our own hi-pass/compressor before trusting either verdict.
  • Recording codec/bitrate — Tone.Recorder's mimeType option is unused (browser default); worth checking if it's limiting quality.
  • Auto level-match — normalize recorded peak/RMS automatically so the volume slider doesn't need manual re-riding per take.
  • Tempo-following playback (time-stretch) — swap Tone.Player for Tone.GrainPlayer, track tempo-at-capture, and recompute playbackRate on tempo change instead of truncating/gapping. Real risk: granular stretching can sound warbly on transient-heavy material or large tempo swings.
