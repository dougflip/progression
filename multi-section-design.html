Multi-Section Progressions

Let users define multiple progressions (song sections like verse, chorus, bridge) within a single setup, and optionally sequence them into an arrangement.

This doc = the current plan. Earlier explorations, deferred features, and the design history live in multi-section-archive.html.

Decisions baked in

  • Max 6 sections.
  • Sections referenced by 1-based number. No naming for v1.
  • Sections stored as array of objects (not bare strings), so future meta (name, per-section style/voicing override) can be added without a data-shape break.
  • No per-section overrides for v1 — global key/style/bass/voicing/tempo only.
  • Reorder via Setup sheet (up/down buttons). No drag.
  • All editing in Setup: add / edit / reorder / delete sections and the arrangement all happen in the Setup drawer. The main screen is playback + jump navigation only — no inline add/edit.
  • Unified playback model: the arrangement defines play order. Empty = implicit linear list (all sections once, in order, looped). A single token (e.g. 2) loops just that section. Multiple tokens = the full song.
  • Advance mode: an Auto / Manual toggle. Auto plays through the sequence and loops; Manual loops the current section until you tap Next. Same boundary mechanism drives both — the default action at each section end is Advance (Auto) or Repeat (Manual).
  • Auto loops forever: this is a practice tool, not a performance player — no play-once/stop.
  • Play starts at the currently-selected section, so you can drop in anywhere — not forced to section 1.
  • Cycle 4ths/5ths transposes each time the active loop restarts: per full-sequence pass in Auto, per section-loop in Manual (preserves today's single-progression key drill).
  • Manual jumps are scheduled: tapping a scrubber segment while playing waits for the current section to finish, then jumps there. The target pulses until the jump fires.
  • Transition signal — deferred (post-v1): first pass changes sections at the boundary with no cue. Design parked in the archive.
  • URL: repeated section params + arrangement + activeSection.
  • No back-compat with the current progression param.

First-pass scope

In v1: multiple sections (max 6); the arrangement (empty = linear list, single token = loop one, multi = song); Auto/Manual advance with Next + tap-to-jump (switch at the section boundary); Auto loops forever; Play starts at the selected section; Cycle 4ths/5ths; Layout D scrubber; the Setup editor.

Deferred to later: the transition signal (visual cue / count-in / drum fills); per-section overrides (style/key/voicing); section names.

Data model

// Replaces single `progression` field on state.
sections: [
  { progression: 'I vi ii V' },     // section 1
  { progression: 'I V vi IV' },     // section 2
  { progression: 'vi IV I V' },     // section 3
  // Future fields (not in v1): name, voicing override, style override, bars override.
]
arrangement: '1 2 1 2 3 2'   // space-separated 1-based refs. Empty = all sections in order (implicit linear list).
activeSection: 1             // 1-based; section shown when stopped + focused in Setup. Play starts here.
advance: 'auto'              // 'auto' = play through & loop forever; 'manual' = hold current section, advance on Next.
cycle: 'none'                // 'none' | '4ths' | '5ths' — transposition; advances each time the active loop restarts.

Object wrapper costs almost nothing now (one extra .progression access in the parse layer) and avoids a breaking shape change later when we add per-section meta.

URL implication: repeated ?section=I+vi+ii+V&section=I+V+vi+IV still works while a section has only one field. When we add per-section overrides we'll likely move to parallel arrays (?sectionVoicing=...) or a JSON-encoded blob. Note it; defer.

Playback modes & use cases

Two things vary independently: what plays (the sequence) and how it advances (on its own, or on your cue).

The sequence

Always the arrangement. Empty = implicit linear list of all sections (1..N); a single token loops one section.

Advance: Auto vs Manual

Auto

At each section's end, advance to the next item. Loops the whole sequence forever (practice tool, not a performance player).

Manual (hold)

The current section loops. It advances only when you tap Next — or tap a specific segment to jump.

One mechanism drives both. At every section boundary the default action is Advance (Auto) or Repeat (Manual); a queued jump (tap Next / tap a segment) overrides the default for the next boundary only. Manual mode just flips the default from "advance" to "repeat."

The use cases

#GoalSetupBehavior
1 Practice one section Manual Sit on a section; it loops. Tap Next when ready → next section (which then loops). Empty arrangement = walk 1→2→3.
2 Play the song Auto + arrangement Plays the arrangement start to finish, loops the whole song forever.
3 Rehearse at your pace Manual + arrangement Walks the song order but holds each section until you advance — "one more time on the bridge" during band practice.
4 Drill in all 12 keys Cycle on See below — Manual climbs per section-loop, Auto climbs per song-pass.

Where Cycle 4ths / 5ths fits

Cycle adds a key transposition that advances each time the active loop restarts. The "active loop" depends on the advance mode:

  • Auto + Cycle — active loop = the whole sequence. Each full pass plays in the next key (up a 4th/5th), 12 keys, then wraps. → "Play my whole song in every key."
  • Manual + Cycle — active loop = the current section. Each repeat bumps the key. → "Drill this ii–V–I around the circle." This is exactly today's single-progression behavior, preserved.

Spelling and octave-clamping unchanged (flats ascending 4ths, sharps ascending 5ths, pitch clamped so it doesn't climb into screech range).

Controls this implies

  • An Auto / Manual toggle (readout pill, or Setup).
  • A Next action in Manual mode — big and reachable. Prime candidate for the planned foot pedal: hands stay on the instrument, foot taps to advance.
  • Optional Previous / Restart section.
  • Tap-to-jump works in both modes as an override.

Interactions

Navigating on the main screen

  • While stopped: the chord strip shows the currently-selected section. Play starts there, so you can drop in anywhere. Tapping a scrubber segment selects/previews that section.
  • While playing: tapping a scrubber segment schedules a jump to that position. The target pulses until the current section finishes, then playback jumps there and continues from that token. No inline editing on main — all in Setup.

Scheduled jump / advance

  • Tap target → set pendingJump (arrangement-token index) and paint it with the queued pulse.
  • At the next section boundary, the Tone callback consumes pendingJump: moves the playhead to that token and clears the pulse.
  • If no jump is queued, the boundary default applies: Advance (Auto) or Repeat (Manual). The Manual Next control just queues "advance to the next token."
  • Tapping a different target before the jump fires replaces pendingJump (only one queued at a time). Tapping the current/queued item cancels.

Editing a section

  • Live-rebuild the Tone.Part when the currently showing/playing section's text changes (same debounce as today).
  • Edits to other sections only matter when playback enters them. Don't rebuild until needed.

Adding / deleting / reordering

  • Add: appends an empty section at the end. Disabled at 6.
  • Delete: removes section, shifts numbers down, removes orphan arrangement refs, decrements refs above.
  • Reorder: swap with neighbor; remap arrangement refs to match.

Arrangement parsing

  • Tokens = bare integers, 1..N where N = current section count.
  • Out-of-range or non-numeric tokens: silently dropped.
  • Cap: 16 tokens.

Code touch points

  1. State: replace progression with sections[], arrangement, activeSection, advance in currentState(), applyState(), DEFAULTS.
  2. Parse layer: parseProgression(input, key, bars) stays. Add parseArrangement(arrangementStr, sectionCount) → array of 1-based ints (clamped to 16; invalid/out-of-range dropped silently). Add buildSongChords(sections, arrangement, key, bars) that:
    • If arrangement empty: concatenate every section in order (implicit linear list).
    • Otherwise: for each token, parse sections[tok-1].progression and concatenate, tagging events with section index + arrangement-position index.
    Note the .progression accessor — only place the object-vs-string shape leaks into the code.
  3. Advance + scheduled jump: state vars advance ('auto'|'manual') and pendingJump (arrangement-token index). In the Tone.Part section-start callback: if pendingJump is set, reseat the playhead there; else Advance (auto) or Repeat (manual). Manual Next sets pendingJump to the next token.
  4. Start position: Transport starts at the bar offset of activeSection's position, not always 0.
  5. Build layer: buildPart() takes the concatenated chord list. Event payload gains sectionIndex + arrangement-position index.
  6. Cycle laps: a lap = full arrangement (Auto) or current section (Manual) transposed. Same logic as today, applied to the active loop unit.
  7. Voice-leading: continuous across section boundaries within a lap. voice-lead-loop resets at lap boundary, not section boundary (unchanged).
  8. UI scaffolding:
    • New bottom scrubber bar (Layout D) rendering the arrangement (or implicit linear list when empty).
    • Auto/Manual toggle + Manual Next control on the main screen.
    • Setup sheet: replace the single progression input with a sections list + arrangement input.
    • renderChips() renders chips for the current section only.
    • Tone.Draw callback sets the current scrubber position from sectionIndex; re-render chips when the section changes.
  9. URL state: ?section=I+vi+ii+V&section=I+V+vi+IV&arrangement=1+2+1+2&activeSection=1&advance=auto. Use params.getAll('section').
  10. Presets: currentState() now snapshots the new shape — presets just work.

Resolved

  • Bad arrangement token → silently dropped.
  • Arrangement cap → 16 tokens.
  • Tap segment to jump → schedules a jump; target pulses until reached.
  • Section switch timing → at next section boundary (not bar).
  • Editing location → all section + arrangement editing in Setup. Main screen is playback + jump only.
  • Empty arrangement → implicit linear list of all sections, in order. Loop one section via a single-token arrangement.
  • Layout chosen → Layout D (bottom transport scrubber).
  • Auto end behavior → loops the song forever (practice tool, not performance).
  • Start position → the currently-selected section, so you can jump in anywhere.
  • Transition signal → deferred to a later pass; v1 changes sections at the boundary with no cue.
